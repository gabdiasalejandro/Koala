### Koala

DSL for generating diagrams from structured text.

The goal is to provide a syntax simple enough to be written by humans or
generated by LLMs, while also having an architecture that can grow
toward more layouts, more styling rules, and more semantic domains.

Base Syntax

Nodes are defined using hierarchical numbering:

-   1, 2, 3 represent roots.
-   1.1, 1.2 represent children of 1.
-   1.1.1 represents a child of 1.1.
-   0 can be used as an optional super-root.
-   If 0 exists, then 1, 2, 3 and other first-level nodes automatically
    become children of 0.
-   If 0 does not exist, nodes 1, 2, 3 continue behaving as independent
    roots.

An explicit relation can also be declared using -> or →. If a line does
not include ->, the hierarchical connection has no label.

Example:

    1 DSL Platform
    Defines the main concept.

    organizes -> 1.1 Core
    Contains parser, models and validations.

    renders -> 1.2 Render
    Generates SVG.

The text under each heading is interpreted as the node body.

kind:: Syntax

There is an optional semantic prefix used to mark the node type. This
prefix always appears at the beginning of the line, before the relation
if one exists:

    hl:: 1.2 Users Table

This produces a node with kind=hl, intended to highlight key concepts at
the rendering layer.

It can also be combined with relations:

    hl:: contains -> 1.2.1 Foreign Key

Recommended Example

The main project example is in mocks/concepts.txt.
This file is designed to work reasonably well with tree, synoptic,
synoptic_boxes, and radial.

Usage

The main entry point is main.py, which exposes a simple CLI.

Examples:

    ./.venv/bin/python main.py --layout tree
    ./.venv/bin/python main.py --layout synoptic
    ./.venv/bin/python main.py --layout synoptic_boxes
    ./.venv/bin/python main.py --layout radial

It also supports:

    ./.venv/bin/python main.py --layout radial --input mocks/concepts.txt --output-dir output
    ./.venv/bin/python main.py --layout tree --theme terracotta

Available parameters:

-   --layout: tree, synoptic, synoptic_boxes, radial
-   --input: .txt or .docx file
-   --output-dir: directory where results are generated
-   --theme: color preset registered in render/defaults.py
-   --typography: typography preset registered in render/defaults.py

Compilation generates an SVG:

-   output/concept_map_<layout>.svg

Supported Layouts

Currently there are four layouts:

-   tree: vertical top-down arrangement, useful for classic hierarchies.
-   synoptic: synoptic chart style using brackets, without boxes and
    without relation labels.
-   synoptic_boxes: column-based arrangement with boxes and relation
    labels.
-   radial: mind-map style layout, with the root at the center and
    branches distributed around it.

In radial:

-   The root is placed at the center.
-   Root children are distributed angularly like a star.
-   Deeper branches determine the total radius of the diagram.
-   The engine attempts to avoid overlaps by calculating radial and
    tangential separation.

Architecture

The project is divided into three layers:

-   core/: parsing, DSL models, and file loading.
-   layout/: geometric computation of nodes and edges.
-   render/: translation of the geometric scene into SVG.

Responsibility separation:

-   core understands the language.
-   layout decides positions according to the layout type.
-   render draws the scene and applies theme, typography, and final
    output.

layout Folder

The layout/ folder is organized to grow without duplication:

-   models.py: typed structures such as LayoutBox, LayoutEdge, and
    LayoutScene.
-   shared.py: shared utilities for measurement, text, and geometric
    helpers.
-   tree_layout.py: top-down layout.
-   synoptic_layout.py: synoptic layout using brackets.
-   synoptic_boxes_layout.py: synoptic layout with boxes.
-   radial_layout.py: mind-map style layout.
-   registry.py: registry of layout engines.

Each engine returns a LayoutScene containing:

-   boxes: nodes already measured and positioned.
-   edges: edges with explicit geometry.

This allows renderers to remain independent from the specific math of
each layout.

Render Layer

The rendering engine is divided as follows:

-   scene.py: combines presets, layout, and viewport into a
    RenderContext.
-   defaults.py: registry of themes, typographies, and layout profiles.
-   viewport.py: fits the scene to the page size.
-   geometry.py: shared geometric helpers for drawing.
-   svg_canvas.py: concrete drawing of nodes, edges, and labels.
-   svg_render.py: public entry point for SVG rendering.

Visual configuration remains centralized in defaults.py, through
registries:

-   THEMES: reusable color palettes
-   TYPOGRAPHIES: reusable typography presets
-   LAYOUT_RENDER_PROFILES: default combinations per layout

This makes it easier to add new typography and color configurations
without modifying the rendering pipeline.

Current Status

It is already a useful tool for:

-   prototyping concept maps from text
-   generating visual variants of the same content
-   testing semantic styles by node type
-   experimenting with different layouts without rewriting the parser

There is still room for refactoring and polishing, especially in:

-   finer radial compaction
-   richer syntax rules
-   domain-specific styling
-   stronger DSL validations
-   automated visual tests
